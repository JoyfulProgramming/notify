Notify MVP

The smallest system that proves the architecture

A minimal but real pipeline that accepts a notification from an app, routes it through Google Cloud Pub/Sub, evaluates it against user-defined rules, and delivers matched notifications to the user via Server-Sent Events. It also supports basic rule management (create, list, delete) via an HTTP API.

What the MVP supports

  1. Accepts a notification from an app (simulated via HTTP POST — no Android required yet)
  2. Routes it through Google Cloud Pub/Sub
  3. Evaluates it against user-defined rules (filter-service + rule-api)
  4. Delivers matched notifications to the user via SSE
  5. Supports basic rule management (create, list, delete) via HTTP API

This is real software, not a prototype. The conserved layer built here is the same one that survives into v2, v3, and beyond — every component must pass the deletion test: can it be completely regenerated from its one-sentence spec plus the shared invariants?

Section 3

Bounded contexts & pace layers

The components are deliberately separated by the Pub/Sub boundary. A developer who has never seen the filter-service code can replace it entirely — as long as the replacement subscribes to notifications.captured and publishes to notifications.matched or notifications.discarded according to the same contracts.

The same word — Notification — appears in all three contexts. The Pub/Sub topic is what makes the meaning unambiguous:

  • notifications.captured — a Capture-context Notification: what an app generated
  • notifications.matched — a Delivery-context Notification: cleared by the Matching context
  • notifications.discarded — a Notification that the Matching context ruled out
Why rule-api belongs in Matching

The Rule is the core aggregate of the Matching context. filter-service evaluates it; rule-api manages its lifecycle. Placing rule-api in Delivery would split the Rule aggregate's ownership across contexts.

Why filter-service isn't a pure context mapper

It does translate at the Capture boundary, but it also owns evaluation logic using Matching's Rule aggregate — making it a full member of the Matching context, not a stateless translator.

Pace layers

Layers that change at different speeds. The slow layer is the conserved boundary; almost nothing in this plan touches it once code-writing begins.

🐢 Slow layer — almost never changes

  • pkg/contracts/notification.go — notification.v1 schema
  • pkg/contracts/rule.go — Rule schema
  • pkg/contracts/events.go — rule-changed event schema
  • Pub/Sub topic names — notifications.captured, notifications.matched, notifications.discarded, rules.changed

🚶 Mid layer — changes monthly (the services)

  • cmd/ingestor/ — HTTP → Pub/Sub (Capture)
  • cmd/filter/ — Pub/Sub → rule evaluation → Pub/Sub (Matching)
  • cmd/rules/ — Rule CRUD HTTP API (Matching)
  • cmd/deliver/ — Pub/Sub → SSE (Delivery)

🏃 Fast layer — changes weekly or daily

  • web/ — minimal HTML/SSE page
  • cmd/deliver/ SSE handler — adapts as browser requirements change

Section 5 · The conserved layer

pkg/contracts

These structs are the slow layer. Changing them requires a new schema version.

notification.v1

type Notification struct {
    ID              string            `json:"id"`               // UUID v7, dedup key
    UserID          string            `json:"user_id"`          // set by ingestor; trusted downstream
    SourceApp       string            `json:"source_app"`       // e.g. "com.whatsapp"
    SourceAccount   string            `json:"source_account"`   // account within app; "" = unspecified
    SourceID        string            `json:"source_id"`        // item id in source system; "" = unavailable
    SentBy          string            `json:"sent_by"`          // phone/email/username/bot id; "" = unspecified
    SentIn          string            `json:"sent_in"`          // group/channel/thread; "" = direct/none
    Title           string            `json:"title"`            // may be empty
    Body            string            `json:"body"`             // may be empty
    DeviceID        string            `json:"device_id"`        // UUID identifying source device
    DeviceTimestamp time.Time         `json:"device_timestamp"` // when it appeared on device
    ReceivedAt      time.Time         `json:"received_at"`      // when this system first ingested it
    Metadata        map[string]string `json:"metadata"`         // arbitrary k/v; no service may fail on unknown keys
}
  • ID is assigned once at ingestion and never changes — the dedup key for the entire system.
  • UserID is set by the ingestor from the authenticated token. No service may accept an empty user_id.
  • SourceAccount + SourceApp together uniquely identify the origin.
  • SentBy / SentIn enable the full specificity rule set (sender+group) in v2 — the fields exist now so the schema doesn't need versioning later.
  • No service may error or panic on an unknown metadata key.
  • Adding optional fields is backward-compatible. Removing/renaming fields requires notification.v2.

Section 5 · Conserved decision

Authentication architecture

Decided in the conserved layer because the pattern affects user_id on every Notification and the delivery-service fan-out model. Getting this wrong later requires a schema migration.

The pattern: authenticate once at the boundary, propagate identity on the message.

Delivery-service fan-out model (locked in now)

When multiple users are connected, delivery-service must not fan out every notifications.matched message to every client. The correct approach is Pub/Sub message filtering on the user_id attribute — each user's SSE session uses a subscription filtered to attributes.user_id = "<id>". This scales without per-user topics.

notifications.matched topic subscription: deliver-user-abc123 filter: attributes.user_id="abc123" subscription: deliver-user-def456 filter: attributes.user_id="def456"

The ingestor must set user_id as both a JSON field on the message body and as a Pub/Sub message attribute when publishing — this is a publishing contract, not just a schema contract.

MVP implementation

Trivially simple, architecturally complete:

API key in Authorization header
  → static map: {"local-key": "local"}
  → user_id = "local"

The infrastructure is real (middleware validates the header, user_id is stamped, subscriptions are filtered) but the implementation is a one-liner. Swapping in JWT validation changes only the token-validation function — nothing else in the architecture moves.

Section 6 · The constitution of the system

System invariants

These must hold across any implementation, in any language, across any regeneration cycle. Every contract test and property test maps to one of these.

INV-1 — Nothing is silently lost

A notification that enters the Capture context is never silently lost. It is either present in notifications.matched, present in notifications.discarded, or causes an explicit error. It is never simply absent.

INV-2 — Rules are the single source of truth

No implementation may place a notification on notifications.matched unless at least one rule matches it.

INV-3 — At most one delivery per client per session

A notification in the Delivery context is surfaced at most once per SSE client per session. Duplicates are suppressed by id.

INV-4 — Deterministic routing

The filter-service (Matching BC) is deterministic: given the same Capture-context notification and the same set of active rules, it always produces the same routing decision.

INV-5 — Rule changes apply prospectively only

A rule change takes effect for all future notifications immediately. It does not retroactively promote already-discarded notifications into the Delivery context.

INV-6 — Any match delivers; specificity is reserved for v2

In v1, all rules deliver. If any rule matches, it is delivered. Multiple matching rules cannot conflict — they all say the same thing, so specificity resolution is not needed in v1.

Specificity rules are documented here for v2, when DISCARD (exception) rules introduce conflicts:

  1. More fields populated = more specific: SourceApp+SourceAccount+Title > SourceApp+SourceAccount > SourceApp+Title > SourceApp only > all empty
  2. Within the same field: exact > pattern > empty

These two dimensions combine. No user-assigned priority number ever exists — specificity is always derived from the rule's shape alone.

Capture bounded context

notification-ingestor

Accepts a JSON notification body via POST /notifications, authenticates the caller and stamps user_id, assigns an id (UUID v7) and received_at if absent, validates against notification.v1, and publishes to notifications.captured with user_id as a message attribute.

Provenance

Why it exists

The entry point for the Capture bounded context and the single auth boundary. Separating this from filter-service means filter-service can be regenerated freely without risk of losing or mangling incoming notifications.

Rejected alternatives

Filter directly on write: couples validation to business logic, increasing blast radius on regeneration. Let clients publish directly to Pub/Sub: removes the validation boundary entirely.

Active assumptions

A single instance is sufficient for MVP. HTTP is the ingestion protocol (not gRPC, not Android Intents). id uniqueness is enforced by dedup, not a uniqueness constraint.

What would make this wrong

If the Android listener must publish directly to Pub/Sub (bypassing the ingestor), the validation contract needs to live elsewhere — perhaps a Pub/Sub schema registry.

Matching bounded context

filter-service

Subscribes to notifications.captured, loads the owner's rules from the rule store, and publishes the notification to notifications.matched if any rule matches, or notifications.discarded if none match.

Behavioral examples

Notification matching a rule is delivered
  1. Given the user has a rule {source_app: "com.whatsapp"}
  2. When a notification with that source_app arrives on notifications.captured
  3. Then within 3s it appears on notifications.matched and not on notifications.discarded
No matching rule → discarded
  1. Given no rules exist for source_app "com.instagram"
  2. When such a notification arrives
  3. Then it appears on notifications.discarded only
Catch-all rule (all fields empty) matches anything
  1. Given a rule {source_app: ""}
  2. Then any notification appears on notifications.matched
Pattern rule matches on glob
  1. Given a rule {source_app: "com.google.*"}
  2. Then "com.google.gmail" is filtered in; "com.whatsapp" is discarded
Title pattern narrows matching
  1. Given a rule {source_app: "com.google.gmail", title: "*invoice*"}
  2. Then "Your invoice is ready" is filtered in; "Newsletter: weekly digest" is discarded
Rule change takes immediate, non-retroactive effect
  1. Given a notification was discarded before a matching rule existed
  2. When the rule is added and a second, identical-shaped notification arrives
  3. Then the second is filtered in, and the first remains discarded — no retroactive change

func TestContract_RuleChangeAppliesToFutureOnly(t *testing.T) {
    clearAllRules(t)
    id1 := publishViaHTTP(t, Notification{SourceApp: "com.slack", Title: "Message from Dave"})

Provenance

Why it exists

The core evaluation service of Matching, and the boundary adapter at the Capture→Matching crossing. Filtering happens in the cloud so rule changes apply to all devices simultaneously and the audit log is centralised.

Rejected alternatives

On-device filtering only: rules diverge between devices. Filter inside the ingestor: prevents replaying historical events against new rules (a v2 capability). Filter inside delivery: routing and delivery have different failure modes.

Active assumptions

~50 active rules max in MVP. Specificity derives from rule shape alone. Evaluation is stateless. SQLite rule store is local to filter-service in MVP.

What would make this wrong

Retroactive rule application would need a replay capability. Thousands of rules would need an indexed evaluation approach.

Provenance

Why it exists

Rules are the core aggregate of Matching. They must be manageable without restarting any services. This is the single point of mutation, so audit events are always emitted and filter-service can stay stateless.

Rejected alternatives

Config file: can't change rules at runtime without redeploy. filter-service owns mutation: couples CRUD and evaluation logic, which change at different times.

Active assumptions

user_id = "local" for MVP. SQLite is sufficient; a Postgres swap is a mid-layer change, not a conserved-layer one.

What would make this wrong

Concurrent writers would hit SQLite's single-writer bottleneck — Postgres + advisory locks, or an event-sourced rule store, would be needed.

delivery-service

Subscribes to notifications.matched, deduplicates by id per connected client, and streams matching notifications to all active SSE connections as JSON-encoded events.

Provenance

Why it exists

The outward face of Delivery — takes notifications that cleared Matching and surfaces them to the user. A fast layer that adapts to the delivery channel (SSE today, FCM/WebSocket later) as an addition, not a change.

Rejected alternatives

filter-service publishes directly to SSE: couples a stateless cloud process to stateful client connections. Deliver from the ingestor: the ingestor shouldn't know about connected clients.

Active assumptions

SSE is the only channel in MVP. Each connection authenticates once at upgrade. Dedup is tracked in memory per connection. Disconnected clients may miss notifications — replay is v2.

What would make this wrong

Guaranteed delivery across reconnects would need a durable per-client queue (Redis, or a Pub/Sub subscription per client).

Section 12

Implementation order

Follow the Phoenix sequence: invariants and contracts first, code last.

Pre-code (days 1–2)

  • Review and agree on the plan document. Revise until it feels right.
  • Set up the development environment.
  • Commit pkg/contracts/ — the slow layer — before anything else.
  • Write the evaluation test files as failing tests — the acceptance criteria, before any service code.

Week 1 — Core pipeline

  • filter-service: pure function (Notification, []Rule) → matched | no match. Pass contract tests.
  • notification-ingestor: HTTP handler + Pub/Sub publisher, wired to the local emulator.
  • rule-api: SQLite-backed CRUD + rule-changed events; filter-service reflects changes without restart.

Week 2 — Delivery and end-to-end

  • delivery-service: Pub/Sub subscriber + SSE handler to a minimal HTML page.
  • End-to-end test: POST /notifications → browser receives SSE event.
  • All contract tests green. All property tests green.

Week 3 — Operational readiness

  • Live evaluation: delivery latency from device_timestamp to SSE receipt, logged.
  • Dead-letter logging: anything on notifications.discarded logged with reason.
  • Duplicate delivery tracking: log any SSE event delivered more than once per client.
  • docker-compose up starts the full system in one command.

The n=1 test at day 14: could a new Go developer, given only the specs, invariants, and evaluations (not the code), regenerate the filter-service from scratch, run it, and pass all contract and property tests? If yes — the MVP is complete. If no — improve the specs, not the code.

Section 16 · Resolve before coding

Open questions

Decisions that affect the conserved layer. Get them right before building.

1. Topic naming convention — flat kebab-case (notification-raw), dot-namespaced hierarchy (notify.notification.raw), or DDD-style past-tense events? Whatever we choose becomes a conserved boundary.
Decision: DDD-style past-tense events — notifications.captured, notifications.matched, notifications.discarded, rules.changed. Chosen because it encodes the bounded-context language (capture/match/discard/rule-changed) directly in the topic name, and because renaming filtered → matched removes a real ambiguity in the original draft — "filtered" could be misread as "filtered out," the opposite of what the topic carries.
2. Rule store startup — query SQLite on every message, or maintain an in-memory cache refreshed on rules.changed?
For MVP: querying on every message is simpler. Cache adds complexity but reduces latency.
3. Notification id assignment — does the HTTP client supply the id, or does the ingestor always generate it?
Decision: allow client-supplied IDs; ingestor generates one if absent.
4. Error handling on filter failure — if filter-service can't reach SQLite, nack the message (retry) or ack and route to dead-letter?
Decision needed — these have different retry semantics.
5. SSE reconnection — replay recent notifications or start fresh?
For MVP: start fresh. Replay is v2.
6. Local SQLite file location — shared between rule-api and filter-service, or separate copies synced via rules.changed?
Decision: shared file at a known path in dev. In production this becomes Postgres.
